§ 01 — ArchitectureHow APIs are structured

The Matcher is a Next.js BFF (Backend-for-Frontend). The browser never calls upstream services directly. Sleep Expert actions trigger client calls to /mattressmatcher/api/* (REST BFF) or /mattressmatcher/api/trpc/* (typed RPC). Those routes hold secrets and proxy to upstream systems.

§ 03 — TimingEager vs lazy

The matcher fires a small set of calls eagerly on page load to warm shared caches. Everything else waits for user intent — a quiz submit, a tab switch, a product click.

Page load · eager
  • financing
  • defaultMessaging
  • recommendedAccessories
  • getTryInStoreData (chained)
  • accessorySizeMapping
  • accessoryMapping (GET)
  • cart create/get
Quiz-gated · lazy
  • /api/matches
  • /api/updateTopItems
Tab-gated · lazy
  • tRPC getUnbxdResults per category
  • Cached on return
Interaction-gated
  • cartManagement POST
  • atpCache, atpOOSCache
  • productRatings, productPromos, productXRayDetail
  • productDetails, closeststore, financingLinks

Cache behaviors that skip network calls

Session storage — Returning to Matches tab from PLP restores recommendations without re-calling /api/matches (when recommenderLogId exists and quiz was not just re-submitted).

Memory cache — getNewQuizRecommendedProductsClientSide caches identical payloads for 1 hour.

PLP per-category cache — Switching back to a previously visited category tab restores Unbxd results from in-memory cache.

§ 04 — EnrichmentTry-in-store enrichment (2026-06-25)

After GET /api/recommendedAccessories returns adjustable bases, RecommendedAccessoriesProvider chains GET /api/getTryInStoreData with adjustable-base D365 IDs and the resolved store ID (preferred store → Unify store → global store number). Re-fetches when store context changes. BFF proxies Azure Spring /products?fields=id,tryInStore.

Fallback banner rules (UI only — no extra API)

§ 05 — JourneyFive phases, end to end

Phase A

Initial page load

When a Sleep Expert lands on Matcher, the app loads configuration from SSR and immediately fetches shared catalog and commerce data needed across quiz, matches, and PLP views.

Step-by-step flow

  1. Browser navigates to /mattressmatcher?StoreNumber=…&SleepExpertID=…&UnifySessionID=….
  2. SSR (getServerSideProps on pages/index.tsx) injects Azure App Config, cart props, store, zip, translations, and feature flags including isNewMatcherEnabled.
  3. React providers mount and fire parallel eager client calls — not gated by quiz progress.
  4. After recommendedAccessories returns adjustable bases, RecommendedAccessoriesProvider chains GET /api/getTryInStoreData when a store ID is available.
  5. Cart initializes: POST /api/createCart if no basket UUID exists, otherwise GET /api/cartManagement.
  6. Quiz UI renders; no recommender call yet.

Phase B

Quiz — no API calls

The Sleep Expert answers questions on one scrollable page. All answers persist locally; no recommender or catalog APIs fire during quiz interaction.

  1. Q1 ("What brings them in today?") gates visibility of remaining questions and header progress tabs.
  2. Each answer change updates dreamhub-quiz context and session storage only.
  3. Footer shows progress (e.g. 6/6 Answered on Primary journey).
  4. View Matches! enables when all required questions are complete.
  5. Clicking View Matches! switches the header to post-quiz mode and mounts the Matches tab, which triggers Phase C.

Phase C · critical path

Matches fetch — top mattress + foundation pairings

After quiz completion, the app requests ranked recommendations from Recommender V2 and maps the top three mattress + foundation pairings for the Matches tab.

  1. useMatchesRecommendations mounts with the Matches tab.
  2. Cache check: If session has a recommenderLogId and the quiz was not just re-submitted → restore from session cache (no network).
  3. Fresh fetch: formatNewRecommendationsPayload() translates quiz answers → MatchesRecommendationsPayload.
  4. Client POST to /api/matches (with optional ?refresh=true).
  5. BFF validates payload (422 if customerResponses.mattressSize missing).
  6. BFF POST to Recommender V2; server-side getRelatedVariants adds foundation accessory mappings.
  7. Response mapped to top 3 pairings via mapNewQuizResponseToMatches.
  8. Fire-and-forget POST /api/updateTopItems records top item IDs.
  9. Unify analytics events fire when quizSubmitSuccess was true.

Phase D

Category PLP tabs — mattresses · foundations · bedding

After matches, Sleep Experts can browse full ranked lists by category. Product data comes from Unbxd via tRPC, not the V1 recommender BFF.

  1. Sleep Expert clicks mattresses, foundations, or bedding.
  2. PlpPageDataProvider checks per-category in-memory cache.
  3. Cache hit: instant restore — no network.
  4. Cache miss: tRPC getUnbxdResults.fetch with category queryType and filters.
  5. Filter chip, sort, or pagination change triggers a new Unbxd fetch.
  6. Matches Only filter slices client-side from cached Unbxd data; may use dual-fetch pattern on "View More Matches".
  7. View All Mattress/Foundation Matches navigates to PLP with matcher-derived sort order.

Phase E

Product detail, cart, and search

Opening a product, checking delivery, and adding to cart trigger enrichment and commerce APIs on demand. The new matcher uses on-demand ATP rather than bulk mattress ATP prefetch on load.

PDP open

  1. Sleep Expert clicks a product card (Matches, PLP, or Search).
  2. PdpProvider.initPdpConfig calls tRPC productDetails.
  3. POST /api/productPromos loads promotional badges.
  4. If ratings not cached → GET /api/productRatings.
  5. On size or zip change → POST /api/atpCache (or atpOOSCache for parcel items).
  6. Try-in-store action → GET /api/closeststore.
  7. Apply-now financing link → GET /api/financingLinks.
  8. X-Ray icon → GET /api/productXRayDetail.

Add to cart
