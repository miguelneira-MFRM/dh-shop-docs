§ 01 — OverviewWhat the app does

The DreamHub Mattress Matcher (also called DH Shop New Matcher) is an in-store tool used by Sleep Experts to guide a customer through a mattress-finding quiz and show ranked product matches — mattresses paired with foundations — with pricing, financing, delivery, and add-to-cart.

The new experience replaces a legacy multi-page quiz (Get Started → separate question pages) with a single-page quiz gated by Q1. Legacy paths still exist behind flags and in context/07-legacy-low-priority.md; new work should target the DH Shop flow only.

§ 02 — StackTechnical stack (what kind of app it is)

This is not a standalone SPA — it is a Next.js React application embedded in the DreamHub / Unify in-store Sleep Expert shell.

This is not a standalone SPA. It is a Next.js (Pages Router) React application served at the /mattressmatcher path, embedded in the DreamHub / Unify in-store Sleep Expert shell. Pages use server-side rendering (SSR) for the initial load, then behave as a client-heavy interactive app for quiz navigation, API calls, and results interactions.

Deployment shape: In production, Matcher runs at https://dreamhub.mattressfirm.com/mattressmatcher (DreamHub subdomain — not the legacy www.mattressfirm.com/mattressmatcher Cypress base URLs). Visiting the DreamHub root auto-redirects to Matcher with query params: StoreNumber, SleepExpertID, UnifySessionID. Authentication is Azure AD / MSAL (Microsoft login). It shares types, storage keys, and cart conventions with the broader Unify/DreamHub ecosystem via MFCL, rather than owning those contracts locally.

New vs old UI code: The DH Shop experience adds NewDreamhubMatcher and NewDreamhubNavigation component trees alongside the existing Results/ and HomeLayout/ modules — both coexist during the migration, gated by isNewMatcherEnabled.

§ 04 — User flowPrimary user flow: Quiz → Matches

Feature: Mattress Matcher Quiz → Recommendations

A Sleep Expert walks the customer through a progressive quiz. Before Q1 is answered, a guidance message is shown and progress tabs are hidden. Answering Q1 reveals the remaining questions on the same scrollable page and activates header progress tabs.

High-level summary: When all required questions are answered (6/6 Answered on Primary), View Matches! calls the Matches API and renders top mattress + foundation pairings.

Step-by-step flow

  1. Land on DreamHub / Matcher — Authenticated session at dreamhub.mattressfirm.com redirects to /mattressmatcher?StoreNumber=…&SleepExpertID=…&UnifySessionID=…. Matcher loads inside the DreamHub shell (left nav, search bar, store badge).
  2. Start quiz — Welcome: "Welcome to MattressMatcher!" plus "Let's help your customer sleep well to live well!" and "Tell us about their needs, and we'll curate the perfect selection together." Q1: "What brings them in today?" Before Q1 is answered, guidance reads: "Select an answer above to reveal the rest of the questions."
  3. Choose journey (Q1) — Answer determines one of three journeys:
    • Primary (standard): multi-select among Bed upgrade / Better sleep / Alleviate pain → 6 progress tabs (why shop → disruptors → size → position → comfort → budget)
    • Guest (guest): Guest bed → 4 tabs (why shop → size → comfort → budget)
    • Child (child): Child's bed → 4 tabs; King sizes excluded
  4. Answer questions — After Q1, progress tabs appear and all remaining questions become visible on one scrollable page (guidance heading disappears). Each question shows an "Example:" coaching prompt for the Sleep Expert; some include View More. Footer shows N/6 Answered progress on Primary. Answers persist in session storage.
  5. Submit — View Matches! enables at 6/6 Answered (no separate Next button).
  6. Resolve init action — On Matches tab mount, resolveMatchesInitAction picks one path (mount-only; tab switches do not re-run):
    • profile-preload — First View Matcher entry with a prior customer-profile recommender log → hydrate from profile cache, no network
    • restore-cache — Session has recommenderLogId and quiz was not just re-submitted → restore from session, no network
    • fetch — Fresh quiz submit or no cache/preload → proceed to API
  7. Build API payload — On fetch path only: formatNewRecommendationsPayload() maps quiz answers to MatchesRecommendationsPayload.
  8. Fetch matches — Client POST to /mattressmatcher/api/matches.
  9. Server processing — /api/matches → external Matches API (V2) + foundation accessory mappings.
  10. Map to UI — mapNewQuizResponseToMatches() → top 3 mattress + base pairings with why-it-works, financing, delivery.
  11. Show matches — Header tabs switch to post-quiz mode: matcher quiz | matches | mattresses | foundations | bedding. Matches tab shows "your customer's Top Matches", a Matcher Summary narrative, 3 pairings with Add Both To Cart, Pair & save financing (via useFinancingOffer + config tiers), optional try-in-store toggle, and View All Mattress/Foundation Matches links.

Dependencies & blockers

  • Azure App Config provides dreamHubQuizConfiguration and feature flags
  • Session storage must be available for answer persistence across refreshes

§ 06 — ResultsResults / matches experience

After View Matches!, the app switches to the new matches view (NewDreamhubMatcher) — not the legacy SummaryPage / results.tsx PLP path (though that code still exists for migration).

Live prod UI (validated 2026-06-04)

  • Header tabs (post-quiz): matcher quiz · matches · mattresses · foundations · bedding
  • Matches tab content:
    • "your customer's Top Matches" heading
    • Matcher Summary — narrative tying quiz answers to recommendations
    • 3 match pairings — mattress + foundation side-by-side, Match badge, add-to-cart, financing, try-in-store, delivery
    • "Why this match works for your customer" — attribute chips (e.g. Cooling, Pressure Relief)
    • Add Both To Cart per pairing; Pair & save combined financing (computed client-side via useFinancingOffer from App Config tiers — distinct from page-load GET /api/financing)
    • Try-in-store toggle — Filters match pairings to in-store-tryable SKUs; may show a fallback banner when fallback foundation SKUs (D365 IDs starting with F) appear in the active slice
    • FirmCare nudge: "Protect your investment. Protect your sleep. Learn More"
    • View All Mattress Matches / View All Foundation Matches buttons

Category PLP tabs (mattresses · foundations): When Try In-Store filter is active and every visible product is online-only for the store, a fallback banner may appear (same helper pattern as matches — UI only, no extra API).

Legacy Results components (src/components/Results/*) power the category PLP tabs below.
