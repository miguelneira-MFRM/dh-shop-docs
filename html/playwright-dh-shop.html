§ 01 — TopologyTwo machines, one bridge

Development happens locally on a named branch per machine. The shared main branch on GitHub is the only place the two worlds meet — never the place where new work begins.

BranchMachinePurpose
mainGitHub (remote)Sync bridge — never commit directly
dhshopworkWork laptopActive development at work
dhshophomeHome PCActive development at home
Treat main like an airport, not a desk. You pass through it; you don't live in it.

§ 02 — Git workflowThe session lifecycle

Every coding session follows the same four-beat rhythm: sync down, commit freely, sync up, switch machines. Skipping any beat is how branches drift.

Phase A · open

Starting a session

Always sync with origin/main before writing any code. Rebasing — not merging — keeps the local branch's history linear on top of whatever the other machine last pushed.

At work
git checkout dhshopwork
git fetch origin main
git rebase origin/main
At home
git checkout dhshophome
git fetch origin main
git rebase origin/main

Phase B · work

During a session

Commit freely on the local branch. These commits stay local until the session ends, so the message style can be as scratch or as polished as the moment calls for.

git add .
git commit -m "your message here"

Phase C · close

Ending a session

Push the work up through main so the other machine can pick it up tomorrow. The pattern is symmetrical — merge local into main, push, hop back to the local branch.

At work
git checkout main
git merge dhshopwork
git push origin main
git checkout dhshopwork
At home
git checkout main
git merge dhshophome
git push origin main
git checkout dhshophome

Phase D · interrupt

Switching machines mid-task

When the work isn't ready to commit but the day is over anyway, stash it. The stash lives only on the current machine — so commit and push if you actually need the changes to travel.

# Before leaving
git stash

# After syncing on the other machine, restore your work
git stash pop

Phase E · recover

If you hit a conflict

Conflicts usually mean both machines touched the same file before syncing. Git will pause the rebase or merge; resolve, stage, and commit.

git status                  # see which files conflict
# open and fix the conflicting files
git add .
git commit -m "resolve merge conflict"

The cardinal rule

Never commit directly to main. It is the bridge between machines, not a workspace. If main accumulates its own commits, the rebase model breaks and merges start producing the kind of tangled history that takes an afternoon to unwind.

§ 03 — Playwright commandsThe CLI, grouped by intent

Five categories cover almost everything: installing, running, viewing results, generating code, and the small set of utilities that don't fit anywhere else.

Installation & setup

# Install Playwright and browsers
npm init playwright@latest

# Update Playwright to the latest version
npm install -D @playwright/test@latest

# Install / reinstall all browsers
npx playwright install

# Install a specific browser only
npx playwright install chromium
npx playwright install firefox
npx playwright install webkit

Running tests

# Run all tests (headless by default)
npx playwright test

# Run tests in UI mode — interactive, with time-travel debugger
npx playwright test --ui

# Run tests headed (watch the browser)
npx playwright test --headed

# Run a specific test file
npx playwright test tests/example.spec.ts

# Run tests matching a title keyword
npx playwright test -g "login"

# Run tests in a specific browser only
npx playwright test --project=chromium
npx playwright test --project=firefox
npx playwright test --project=webkit

# Run tests in parallel with a set number of workers
npx playwright test --workers=4

# Run in debug mode (pauses at each step)
npx playwright test --debug

Viewing results

# Open the last HTML test report
npx playwright show-report

# Show a specific report folder
npx playwright show-report playwright-report

Code generation

# Open codegen to record a new test by clicking through the browser
npx playwright codegen https://your-site.com

# Record into a specific output file
npx playwright codegen https://your-site.com --output tests/new-test.spec.ts

Utilities

# Check installed Playwright version
npx playwright --version

# List all available projects defined in playwright.config.ts
npx playwright test --list

# Show all test names without running them
npx playwright test --list --reporter=list

§ 04 — Config referenceWhat lives in playwright.config.ts

Tests are configured at the project root. The settings below are the ones touched most often — the rest of the file usually stays untouched between sessions.

SettingDescription
testDirFolder where test files live
projectsBrowser targets (chromium, firefox, webkit)
use.baseURLBase URL prepended to all page.goto() calls
use.headlesstrue by default; set false to watch tests run
retriesHow many times to retry a failing test
reporterOutput format — html, list, dot, etc.
